Service brief

Two artifacts. One upstream operating surface.

I work upstream of delivery. The output is not implementation, coaching, or org-change ownership. The output is a written artifact that makes the system legible enough for builders and decision owners to act without drift.

There are two engagement types: a Diagnostic Artifact, which exposes the real operating structure and failure concentration, and a Redesign Artifact, which specifies the boundary and transition sequence that makes change deterministic.

01 // Diagnostic Artifact entry engagement

Expose the real structure behind recurring failure.

£10k / 14 days
upstream written artifact fixed-scope

This engagement is for systems that keep failing in the same way under growth, complexity, or organizational scale. The goal is to make the actual operating grammar explicit.

What it defines

Primitives, invariants, enforcement points, structural mismatches, and the build map implied by the current state.

What you send

The recurring failure pattern, the interfaces involved, and any current artifacts you already have: diagrams, policies, workflows, notes, or internal documents.

What you receive

One written diagnostic artifact you can hand to builders and decision owners. It is designed to clarify structure, not to decorate it.

Included

  • Category truth and system type clarity
  • Invariant definition (fail-closed rules)
  • Enforcement architecture (where rules live)
  • Boundary visibility across teams, services, policies, and contracts
  • A build map for sequence, interfaces, and transition logic

Not included

  • Delivery ownership
  • Implementation execution
  • Org-change programs
  • Coaching or facilitation retainers
02 // Engagement flow how it starts
  • Step 1: Email five lines on what keeps breaking, what is at stake, and the interfaces involved.
  • Step 2: Include any relevant artifacts that reflect the current system state.
  • Step 3: I assess fit and confirm whether the system should begin with diagnosis or move directly to redesign.
  • Step 4: If the fit is correct, I return the artifact in written form for internal use.

03 // Redesign Artifact follow-on engagement

Specify the boundary and transition sequence for change.

£15k–£25k
post-diagnosis written artifact scope-dependent

Range-based pricing depends on boundary width, interface count, and transition complexity.

This engagement follows when the diagnosis is already clear enough that the next requirement is not analysis, but redesign logic. The goal is to define what must change, where, and in what sequence.

What it defines

Target boundaries, transition sequence, interface changes, and the conditions that allow the system to change without collapsing into ambiguity or local drift.

What you receive

One written redesign artifact that specifies the new operating boundary and the order of change required to make implementation deterministic.

When to use it

After diagnosis, or when the failure mode is already obvious and the real bottleneck is lack of upstream redesign structure.

Included

  • Boundary redesign
  • Transition sequence and dependency order
  • Interface and handoff clarification
  • Change logic that builders can execute against
  • Explicit refusal of steps that would create drift or hidden re-coupling

Not included

  • Program management
  • Team delivery leadership
  • Technical build execution
  • Stakeholder alignment workshops
04 // Fit who this is for
  • Systems that keep failing in the same way under scale
  • Teams dealing with recurring boundary confusion
  • Decision owners who need structure, not more interpretation
  • Organizations where “policy says / reality does” has become normal
  • Contexts where implementation is blocked by upstream ambiguity
